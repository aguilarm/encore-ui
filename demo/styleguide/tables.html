Basics

  • Tables have default styles: regular rows have white backgrounds, while headers use white text with a gray background. An additional table-striped class is available for alternate row striping.
  • ENCORE-SPECIFIC: In Encore, we usually use to tables to show data-table related data, such as listing of objects.
  • If you need to show summary-style important pieces of metadata that aren't necessarily related to each other, consider using a metadata component.
  • In Support Services, there have been moments where we have displayed table in a structural format, but we use a different styling. See the Non-Data based Tables section for more information.

Directives

Cog Menus via rxActionMenu

  • See the rxActionMenu directive for implementation and examples.
  • ENCORE-SPECIFIC: In Encore, we usually use cogs if there are actionable items for every row. As an example, most main table listings show cog menus with actionable items, such as Cloud Server pages showing FirstGen/NextGen actions or Cloud Block Storage Volumes.
  • For general advice as to what icons and colors to use, consult the section on icons and colors.
  • There may be cases where an action is not available on a particular row. In this case, those particular links were disabled using the .disabled class rather than not having them appear at all.
  • HEADS UP: In Encore Cloud's servers.less, we added the the actions class to the table header and styled the column at 15 pixels. This has not been implemented for Encore-UI.
  • HEADS UP: We have one design pattern that slightly differs; for data tables that have involved objects that only require deleting such as particular data tables related to LBaaS connection nodes and Support Service user tables, a particular delete-based design pattern was created. See the section on Delete "X" actions.

Table Footer Pagination via rxPaginate

  • See the rxPaginate directive for implementation and examples.
  • ENCORE-SPECIFIC: The pagination is visible in this example to show how mark-up will be implemented. Generally, the pagination directive does not display if there is only one page of data available. We use angular.js to make sure this doesn't display.
  • HEADS UP: It's a known issue that the footer wraps into a second line if the table is 50% of the screen. In cases like those, we've used the display:none CSS property for elements to hide all elements not directly related to pagination. See the LESS of narrow tables used in Support Services for an example of this. Note that this has not been implemented in Encore-UI yet.

Design Patterns

Null patterns

  • ENCORE-SPECIFIC: Consider having a default message if there are no rows for your data table. In Encore, we still include the data table headers to show there would usually be tabular data here.

Non-Data based Tables

  • For Support Services, we use tables as a structural format but use different styling to show larger components of data such as tags, groups, badges and other block level information.
  • As the usage of this is outside of Encore-UI and falls outside the scope of this document, please look at the the staging site and the specific markup and LESS files for more information.

Delete "X" action

  • ENCORE-SPECIFIC: For data tables involving objects that only require deleting (Examples: tables related to LBaaS connection nodes and Support Service user tables) there's a design pattern that replaces the cog with an X.

Status columns

  • ENCORE-SPECIFIC: Outside the scope of this document is the color coding of status columns. There are currently two different design patterns for this: one inside Cloud Servers to display statuses of cloud based objects (example of server markup), the other inside Ticket Queues to show ticket status.
  • As the usage of this is outside of Encore-UI and falls outside the scope of this document, please look at the the staging site and specific LESS files for more information.

Filters

  • ENCORE-SPECIFIC: Tables have also been used for filtering, but because there has not been a common design pattern for this, this is not integrated into Encore-UI either. Examples include:
    • Data based filters, seen in Encore Cloud (Servers, Volumes, Images, Database, Load Balancers) - Example of server markup
    • Ticket Queues (Filtering by ticket queues and status)
    • Billing (Filtering by billing data and transactional status)
  • Please look at the the staging site and specific mark-up and LESS files inside individual repositories for more information.
